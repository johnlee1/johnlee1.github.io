Hana
A social network website. Previously deployed on prayforhana.org. The site is currently not deployed. Check out my github account (johnlee1) to see the UI code and the API code.
Technology Infrastructure
The frontend is an angular 2 app served by nginx. The api backend is a nodejs app using hapi. Both nginx and node live in docker containers that are deployed with aws ecs. The containers live in ec2 instances and are fronted by amazon's elastic load balancer. The nodejs app is connected to a mongodb database provided by mlab. DNS is handled with namecheap and aws route53. Version control is handled with git and github.
Devops and Scalability
I want to create a more automated process for developing, building, and deploying the applications by looking into more options for ci/cd. In regards to infrastructure, currently the project is set up in a way that does not allow the frontend and backend to scale independently of each other since they are both configured to live on the same server instance. In the future it could possibly be beneficial to have a more flexible microservices setup that allows independent horizontal scaling.
Topic and Sentiment Analysis
A feature I'll be looking into adding is the insertion of sentiment-appropriate and topic-appropriate bible verses in the prayer mode slideshow. Once there is enough data (~50,000 prayer posts), I want to train a machine learning model to classify prayer posts based on sentiment (happy, sad, etc) and topic (relationships, missions, etc). This model can then be used to classify future prayer posts. In this way, during a prayer mode slideshow, context-relevant bible verses (pre-selected to be associated with a sentiment/topic) can appear along with the classified prayer posts. Options for creating the model includes using google's prediciton api, TF-IDF kmeans with scikit-learn, and LDA or LSA with gensim. A vocabulary of prayer posts could be too sparse for good classification without a vector representation of words. One challenge is getting from word vectors to a good document representation (doc2vec) that works for clustering as most clustering algorithms require a fixed-length document vector.
Design (UI/UX)
Hana is a web application that provides a platform for sharing prayer requests. The goal is to make an application that connects people through prayer but without replacing real-life community. Hana is designed for those who want to pray. An expected common use case of this application is learning how to pray for others when it is not possible to directly ask them for prayer requests. Through dialogue with several designers at Carnegie Mellon University, we've come with a four-stage plan to build this application: gain a deep understanding of the problem, validate a solution, design the solution, and build the solution. The problem to be solved is "I want to pray for X, but I don't know how to pray for X". Two types of activities were conducted to gain a deeper understanding of this problem. The first activity was to interview people with this problem (user research). The second activity was for me to go and try to find prayer requests for people whom I wanted to pray for. I choose the Nepali refugee community in Pittsburgh and through talking to friends, visiting churches, and visiting the community itself, I learned what it is like to discover prayer requests of people I do not yet personally know. Validating a solution was done by rapid prototyping of the concept (using existing blogging and q/a platforms) to prototype the experience of posting and discovering prayer requests. This allowed us to discover painpoints of using existing platforms and assess the validity of building a new platform. Hana is being designed collaboratively with a user-centric approach through story mapping, empathy mapping, iteration, and user testing. The intended tone to be achieved with Hana can be described with the following: informative, encourages reflection, simple. The ux/ui sholud be designed with these in mind. Building the solution will require communication between designers and developer to ensure that the development is in-align with the design. The biggest lesson I've learned is the importance of understanding the problem space and of validating a solution. Designing and building a solution are more well-defined tasks that I am confident can be done. The harder problems to solve have been understanding the problem and validating a solution. Understanding the problem has been difficult because of the lack of historical data (there are no existing products on the market to learn from) and the inherent difficulty of finding prayer requests of people whom I do not know. Validating a solution has been invaluable because it ensures that the right product is being built. The difficulty of validating a solution has been taking a very large solution space and coming up with reasonable constraints that narrows the solution space to a single web product.
